# W01 — OpenClaw → PIC 2.0 Bridge

This is my own hypothesis, built from InGen's public `/origami.html` disclosure of the PIC 2.0 pipeline (order, per-stage role, design-target latency — see `W01_InGen_NN_Landscape.md` §6) plus my own OpenClaw architecture. It is not confirmed InGen internal design. OpenClaw is a 6-agent system (Planner, Knowledge, Quant, Risk, Report, Critic — one more agent than PIC 2.0 has models), each backed by an independent FastAPI microservice, orchestrated by the Planner over deterministic HTTP calls rather than a trained control policy.

| PIC 2.0 model | Stage role (public) | Closest OpenClaw component | Claim |
|---|---|---|---|
| **HTD-IRL** | Hierarchical task planning | **Planner agent** | Both decompose one incoming request into an ordered set of downstream calls: HTD-IRL turns a goal into a multi-step robot task; Planner classifies intent (DOC_QA / QUANT_QUERY / RISK_QUERY / MIXED_QUERY / report) and routes to the specific agents needed for that intent. Strongest match in the table — same functional slot (decomposition → routing), different substrate (physical sub-tasks vs. software agent calls). |
| **SEOM** | Hardware-enforced safety gate before actuation | **Critic agent** | Both are the mandatory last-stage gate before output leaves the system: SEOM blocks unsafe physical actions via rules trained into the model itself; Critic runs a 5-point checklist (evidence coverage, data freshness, conflict of interest, overstatement detection, action-boundary declaration) and can block report release. Second-strongest match — both are "nothing ships without passing this stage," though SEOM is trained-in (structural) while Critic is a runtime rule check (a real, worth-stating difference, not just a naming coincidence). |
| **STUM** | Uncertainty quantification that gates downstream action | **Risk agent** | Loose match. STUM scores *perceptual* confidence (e.g., "how sure is this sensor reading") and can make the system hedge or escalate. Risk agent scores *financial* uncertainty (max drawdown, beta, scenario stress tests) and outputs a named risk level (LOW/MODERATE/HIGH/CRITICAL) that downstream reporting must respect. Same structural role — "quantify confidence, gate action on it" — but Risk agent's uncertainty is about market exposure, not sensor trust, so this is a functional analogy, not an architectural one. |
| **GRPO** | Learned policy decision (RL fine-tuned) | **Quant agent** — partial, flagged gap | Weakest confident match in the table. Quant agent produces a composite score mapping to a rating (POSITIVE/BALANCED/NEUTRAL/CAUTION) — a *decision output*, which is GRPO's role structurally. But GRPO is explicitly an RL-trained policy (InGen states `N=16, γ=0.985` fine-tuning parameters publicly on the Aido Humanoid page), while Quant agent's scoring is a deterministic weighted formula over technical + fundamental ratios. OpenClaw has no RL-trained decision component anywhere in its stack — this is a genuine capability gap, not just an imperfect analogy. |
| **AMDC** | Real-time sensor drift calibration (14 sensor types, 100Hz) | *No OpenClaw analogue* | I don't think this maps to anything in OpenClaw, and I think that's the correct conclusion rather than a gap to paper over. AMDC exists because physical sensors drift with temperature/vibration/wear over time. OpenClaw's inputs (SEC EDGAR filings, yfinance price data) don't drift in that sense — they're either correct-at-fetch-time or stale, which OpenClaw already handles differently (the Critic's data-freshness check, not a calibration model). This is the clearest evidence that PIC 2.0's six-model design was shaped by embedded/physical constraints that a pure software agent stack never has to solve. |
| **CRL-MRS** | Multi-robot fleet coordination | **Report agent** — partial | Structural echo, not a real match. Both take multiple parallel outputs and merge them into one coherent deliverable (CRL-MRS: multiple physical robots' state/actions → one fleet plan; Report: multiple agents' JSON payloads → one templated markdown report). But CRL-MRS is coordinating *simultaneously acting physical units* toward a shared goal, while Report is a one-shot synthesis step with no coordination problem to solve (nothing OpenClaw does is happening in parallel across independent physical agents). I'd call this "same shape, different problem." |

## What this hypothesis gets right vs. leaves open

Three of six mappings (HTD-IRL, SEOM, STUM) hold up as genuine functional analogues — decomposition, final-stage safety gating, and confidence-gated action are patterns that show up in any multi-stage decision system, physical or not, which is presumably why OpenClaw independently converged on something similar. The other three (AMDC, GRPO, CRL-MRS) expose real gaps rather than forced analogies: AMDC has no OpenClaw counterpart because financial data doesn't drift like physical sensors; GRPO has no counterpart because OpenClaw has no RL-trained policy anywhere in it; CRL-MRS's "many-to-one" shape superficially resembles Report but solves a coordination problem OpenClaw never encounters. I think the honest scope of what OpenClaw actually demonstrates toward PIC 2.0 is: **routing/decomposition and safety-gating transfer directly; uncertainty-gating transfers as a pattern; sensor calibration, RL-trained policy, and multi-agent physical coordination do not exist in OpenClaw today and would be the real net-new skills this internship needs to build** — which is consistent with why Week 2 (edge classifier) and Week 3 (fine-tuning) are scoped as new work rather than restatements of OpenClaw.
